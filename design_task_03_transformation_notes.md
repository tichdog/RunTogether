# Задание 3. Преобразование диаграммы классов в структуру БД

Диаграмма: `design_task_03_database_class_diagram.puml`.

Преобразование выполнено по алгоритму из методического материала:

1. Персистентными выбраны классы-сущности: пользователь, сессия авторизации, тренировка, заявка на участие, уведомление, отзыв, достижение, достижение пользователя, жалоба, предупреждение, событие ленты активности и системная настройка.
2. Неперсистентные граничные и управляющие классы из задания 2 в структуру БД не перенесены.
3. Типы атрибутов заменены на типы PostgreSQL: `BIGINT`, `UUID`, `TEXT`, `INTEGER`, `BOOLEAN`, `TIMESTAMPTZ`, `NUMERIC`, `JSONB`.
4. Простые связи заменены внешними ключами: например, `workouts.organizer_id`, `reports.reporter_id`, `activity_feed.actor_id`.
5. Связь N:N между пользователями и тренировками преобразована в связную таблицу `workout_participants` с составным первичным ключом `workout_id + user_id`.
6. Связь N:N между пользователями и достижениями преобразована в связную таблицу `user_achievements` с составным первичным ключом `user_id + achievement_id`.
7. Композиции преобразованы так, что ключ владельца входит в первичный ключ зависимой таблицы: `auth_sessions.user_id`, `notifications.user_id`, `workout_participants.workout_id`, `reviews.workout_id`.
8. Перечисления из диаграммы классов представлены как `<<enum table>>`. В текущей PostgreSQL-схеме проекта они реализованы через `TEXT` с `CHECK`, но на учебной диаграмме показаны отдельными таблицами согласно алгоритму из методического материала.
9. Все связи на итоговой диаграмме имеют тип зависимости `..>` и подписаны именем поля внешнего ключа.

Примечание: диаграмма задачи 3 показывает результат преобразования по учебному алгоритму. В текущей реализации проекта часть составных ключей заменена на surrogate key `id` плюс уникальные ограничения, что технически эквивалентно для хранения допустимых комбинаций, но в отчете оставлен вариант, который ближе к алгоритму из PDF.
